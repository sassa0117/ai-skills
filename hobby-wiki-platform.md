# ホビーコレクターWiki プラットフォーム構想書

## 1. コンセプト

「アニメ・ホビーグッズ版Discogs」
アイテム軸のWikipedia × Reddit。ユーザーが相場データ・購入記録・レビューを投稿し、コミュニティ駆動の相場データベースを構築する。

### 1-1. なぜこれを作るのか（市場背景）

- 中古せどり情報商材は普及曲線の転換期にある（ロジャースの普及理論）
- 「稼ぐ手段」としてのせどりは飽和 → 「ライフスタイル/ホビー」への移行が進行中
- 日本のアニメ・ホビーグッズの中古市場は巨大だが、相場情報が分散している
- Discogs（レコード）やPrice Charting（ゲーム）に相当するサービスが、日本のアニメ・ホビーグッズにはない
- オークファンは相場ツールだがコミュニティがない。MyFigureCollectionはDBだが価格追跡が弱い

### 1-2. Discogsの実績（参考）— 詳細データ

- 2000年、Kevin Lewandowski（Intel退職DJ）が自宅クローゼットで単独開発。電子音楽メーリングリストの仲間が初期ユーザー
- 本人が200枚入力して概念実証 → 毎月トラフィック倍増
- 2003年に最初のMP試行（$0.50/出品課金）→ 失敗して撤回
- **2005年にMP再ローンチ（出品無料、売上手数料のみ）→ 2年目で130,675人が取引**
- MP開始前からユーザーが「everything in this collection is for sale」のアカウント名でハック販売 → 正式機能化
- 2018年まで**18年間VCなしでブートストラップ**。外部調達はわずか$2.5M
- 現在: 1800万+リリース、年間コレクション登録1.142億件、GMV $253M/年、手数料率9%（推定手数料収入$22-23M/年）
- 累計コレクション登録9.2億件。2026年に10億件突破見込み
- **コントリビューター率10%以下**。アクティブ投稿者~10,000人、モデレーター<500人
- **核心設計: 自分のコレクション管理（＝個人利用）のために登録 → 副産物としてDBが育つ**

### 1-3. さっさの資産が活きる理由

- 毎日の仕入れ活動 = プラットフォームへのデータ投稿（二重投資なし）
- 中古ホビー・アニメグッズの相場観 = 初期データの質を保証
- Xフォロワー6,900人 = 初期ユーザー獲得チャネル
- 既存の技術スタック（Next.js + Prisma）= そのまま開発可能

## 2. ターゲットユーザー

### フェーズ別ターゲット（ロジャースの普及理論に基づく）

| フェーズ | 時期 | ターゲット | 特徴 | 集客方法 |
|---------|------|-----------|------|---------|
| Phase 0 | ローンチ前 | さっさ自身 | 初期データ投入。最低100アイテム | 自分で入力 |
| Phase 1 | 0-3ヶ月 | イノベーター層 | せどらー・ガチコレクター。データ入力を楽しむ。新しいツールに飛びつく | X告知、せどりコミュニティ |
| Phase 2 | 3-12ヶ月 | アーリーアダプター | 相場を知りたいコレクター・出品者。実用価値で使う | SEO、口コミ、Reddit/オプチャ |
| Phase 3 | 1-2年 | アーリーマジョリティ | 一般のフリマ出品者、推し活民。「みんな使ってる」で参入 | プラットフォーム内回遊、SNS |
| Phase 4 | 2年以降 | レイトマジョリティ | 相場を調べたいだけの人。ROM専でも可 | Google検索流入 |

### キャズム（Phase 1→2の溝）を超える条件

- データ量の臨界点（検索して「出てくる」状態）
- SEOで「商品名 + 相場」で上位表示される
- ユーザーが「ここを見れば相場がわかる」と認知する

## 3. 機能設計（MVP）

### 3-1. コア機能（Phase 1で必要）

1. **アイテムページ**
   - 商品名、カテゴリ、画像、説明文
   - ユーザーが作成・編集できる（Wiki式）
   - JAN/型番がある商品は自動紐付け

2. **相場記録**
   - 購入価格、販売価格、日付、状態（新品/中古/ジャンク）
   - 購入元（メルカリ/ヤフオク/実店舗/オフモール等）
   - スクリーンショット添付（任意）
   - 1アイテムに複数ユーザーが記録を投稿

3. **相場グラフ**
   - アイテムごとの価格推移を可視化（Recharts）
   - 平均価格、最高値、最安値

4. **コメント/ディスカッション**
   - アイテムごとのスレッド
   - 「これ偽物多いから注意」「セカストで見つけやすい」等の情報共有

5. **ユーザーシステム**
   - 登録/ログイン（メール or SNS認証）
   - プロフィール（投稿履歴、得意ジャンル）
   - 投稿数によるランク/バッジ（ガミフィケーション）

6. **検索・ブラウズ**
   - キーワード検索
   - カテゴリ別閲覧（フィギュア/プラモ/カード/ゲーム/アパレル等）
   - 新着アイテム、人気アイテム

### 3-2. 後から追加する機能（Phase 2以降）

- タグシステム（作品名、キャラクター名、メーカー等）
- 「ウォッチリスト」（特定アイテムの相場変動を通知）
- マーケットプレイス機能（ユーザー間売買）
- API公開（外部ツール連携）
- 多言語対応（海外コレクター向け）

### 3-3. 昨今のプラットフォームから学ぶUX設計方針

| 参考元 | 学ぶポイント |
|-------|------------|
| メルカリ | 出品の手軽さ（写真撮って即投稿）。相場記録もこの手軽さで |
| TikTok | 無限スクロールのフィード。「最近の相場記録」をフィードで流す |
| note | クリエイター（投稿者）が主役のUI。投稿者プロフィールの見やすさ |
| YouTube | おすすめアルゴリズム。関連アイテムの自動レコメンド |
| Instagram | ビジュアルファースト。商品画像・スクショが映えるレイアウト |
| X | リアルタイム性。「今日の仕入れ」をすぐ投稿できる速度感 |
| Reddit | アイテム単位のスレッド。投票でいい情報が上に来る |
| Voicy | 属人性。「この人の投稿を追いたい」というフォロー導線 |

### UX原則
- **投稿は30秒以内に完了できる**（Fogg行動モデル: Abilityを最大化）
- **見るだけでも楽しい**（フィード＋ビジュアル。ROM専でも滞在する）
- **投稿するとフィードバックがある**（いいね、バッジ、ランクアップ。可変リワード）
- **スマホファースト**（出先で仕入れ→即投稿の導線）

## 4. 技術スタック・アーキテクチャ

### 4-1. 技術選定

| レイヤー | 技術 | 理由 |
|---------|------|------|
| フロントエンド | Next.js (App Router) + React | さっさの既存スキル、SSR/SEOに強い |
| スタイリング | Tailwind CSS v4 | さっさの既存スキル |
| バックエンド | Next.js API Routes / Server Actions | フロントと統合、初期はこれで十分 |
| ORM | Prisma | さっさの既存スキル |
| DB | PostgreSQL（Supabase or Railway） | スケール対応。SQLiteでは複数ユーザーの同時書き込みに弱い |
| 画像ストレージ | Cloudflare R2 | エグレス（転送）無料。S3互換API。画像が増えてもコスト爆発しない |
| 認証 | NextAuth.js (Auth.js) | Google/X/メールログイン対応 |
| ホスティング | Vercel（フロント）+ Railway or Supabase（DB） | Vercelの無料枠 + DB月額$5程度 |
| グラフ | Recharts | さっさの既存スキル（在庫管理アプリで使用済み） |
| 検索 | 初期はPostgreSQL全文検索 → 将来的にMeilisearch | 初期コストゼロ |

### 4-2. スケール戦略

| ユーザー規模 | 対応 |
|------------|------|
| ~100人 | Supabase無料枠（500MB DB + 1GB Storage）で十分 |
| 100-1,000人 | Supabase Pro（$25/月）or Railway（$5/月+従量） |
| 1,000-10,000人 | Cloudflare R2本格運用、CDNキャッシュ、DB最適化 |
| 10,000人以上 | 専用インフラ検討。ただしここまで来たら収益でカバーできる |

## 6. マネタイズ戦略

### Phase 1-2（初期〜成長期）: 収益化しない
- ユーザー数とデータ量の蓄積に集中
- 広告もマーケットプレイスも入れない
- **コミュニティの信頼構築が最優先**

### Phase 3（拡大期）: 収益化開始
| 手段 | 想定収益 | 備考 |
|------|---------|------|
| 広告（Google AdSense等） | 月1-5万円 | PV依存。SEOが効き始めたら |
| プレミアム機能（月額） | 月額500-980円 | 価格アラート、詳細分析、CSV出力 |
| アフィリエイト | 成果報酬 | メルカリ/ヤフオクへのリンク |

### Phase 4（成熟期）: 本格収益
| 手段 | 想定収益 | 備考 |
|------|---------|------|
| マーケットプレイス手数料 | 取引額の5-8% | Discogs式。ユーザー間売買 |
| 法人向けAPI | 月額課金 | リサイクルショップ向け相場データ |
| スポンサーシップ | 案件単位 | 駿河屋、まんだらけ等 |

## 7. AIスパム対策（懸念への対応）

### 想定リスク
- AIで大量の虚偽相場データを投稿される（インプレゾンビ的）
- 収益化後にアフィリエイトスパムが増える

### 対策（段階的に導入）

| Phase | 対策 |
|-------|------|
| Phase 1 | 招待制 or 承認制で初期ユーザーを絞る。信頼できるコントリビューターだけ |
| Phase 2 | 投稿にスクリーンショット添付を推奨（エビデンスベース） |
| Phase 2 | ユーザーの信頼スコア（投稿数×いいね率）で重み付け |
| Phase 3 | レートリミット（1日の投稿上限） |
| Phase 3 | コミュニティモデレーション（通報→非表示） |
| Phase 4 | AI検出（テキストの不自然さ、投稿パターン分析） |

### 初期は「招待制」が最強
- Phase 1はイノベーター層だけ入れたい
- 招待コードでアクセスを制限 → データの質が担保される
- 「選ばれた人だけ使える」= 希少性（Cialdini）が働いてバイラルしやすい

## 8. 成長戦略（普及理論 × マーケティングサイエンス）

### Phase 0: 種まき（ローンチ前）
- さっさが自分で100アイテム+相場データを投入
- 「検索して何かが出てくる」状態を作る（空のプラットフォームに人は来ない）
- Xで「こういうの作ってます」のチラ見せ投稿
- ツァイガルニク効果: 完成前に見せる→「いつ使えるの？」の期待を作る

### Phase 1: イノベーター獲得（0-3ヶ月）
- 招待制で20-50人のコアユーザーを確保
- Xのせどらー・コレクターコミュニティに直接声掛け
- IKEA効果: 「一緒に作ってる」感覚→初期ユーザーの愛着
- 返報性: 「あなたのデータが他の人の役に立つ」→貢献動機
- バッジ・ランク: 投稿数で称号が変わる（可変リワード）

### Phase 2: アーリーアダプター（3-12ヶ月）
- SEO対策: 「商品名 + 相場」で上位を狙う
- 社会的証明: 「〇〇件のアイテム、〇〇件の相場データ」を表示
- 招待制を解除、一般公開
- ユーザー生成コンテンツがSEOコンテンツになる好循環

### Phase 3: キャズム越え（1-2年）
- 「ここを見れば相場がわかる」のポジション確立
- バンドワゴン効果: 「みんな使ってる」
- 収益化開始（プレミアム機能、広告）

## 9. 深層分析（2026-03-07 追加）

### 9-1. コールドスタート問題の詳細対策

**Andrew Chenのアトミックネットワーク理論適用:**
- アトミックネットワーク = 自立して成長できる最小のネットワーク
- Zoom: 2-3人、Slack: 5-10人、Tinder: 500人/キャンパス、Airbnb: 100リスティング/都市
- **本プラットフォーム: 1カテゴリ（例: ワンピースフィギュア）に50人のアクティブユーザー + 500アイテム**
- 全カテゴリを一度にやらない。1カテゴリの「アトミックネットワーク」構築が最優先

**90-9-1ルール（Nielsen）の現実:**
- 10,000人の登録ユーザーがいても、活発な投稿者は100人程度
- その100人のうち10人が全体のデータの大半を作る（Wikipedia: 最もアクティブな1,000人=0.003%が編集の2/3を担当）
- **→ 最初の10人のスーパーコントリビューター確保が生死を分ける**

**Discogsの核心設計からの学び:**
- 「貢献してください」と言わない。「自分のコレクションを管理する」ために使う → 副産物としてDBが育つ
- Collection + Wantlistで「何を持ってるか」「何が欲しいか」を把握 → マーケットプレイスはその自然な接続

### 9-2. ユーザーがデータを投稿する動機（UGC研究に基づく）

| 動機 | 分類 | 設計への反映 |
|------|------|-------------|
| 自分のコレクション整理 | 自己利益 | コレクション管理ツールとしての完成度を上げる |
| 利他主義 | 内発的 | 「あなたのデータが○人の役に立ちました」表示 |
| 互酬性 | 外発的 | 「データ共有で他のデータも見られる」設計 |
| コミュニティ帰属 | 内発的 | フォーラム、コメント、イベント |
| 名声・認知 | 外発的 | ランキング、称号、プロフィール表示 |
| 楽しさ | 内発的 | ガミフィケーション、バッジ、レベルアップ |

**重要: 金銭的報酬はトップコントリビューターのモチベーションを下げる（自己決定理論）**
- 「金のためにやってる」と感じた瞬間、内発的動機が消える
- 初期コントリビューターには「先行アクセス」「称号」「コミュニティの中心的存在」で報いる

### 9-3. Googleで調べるのと何が違うのか

| Google検索 | 本プラットフォーム |
|-----------|------------------|
| 情報が分散（メルカリ、ヤフオク、ブログ、Twitter） | 1箇所に体系化 |
| 最新の取引価格が出にくい | リアルタイムの相場推移グラフ |
| 同じ商品の「状態別」価格が見えない | 状態（新品/中古/ジャンク）別の価格帯 |
| 購入元（店舗/フリマ/オークション）の情報がない | 購入チャネル別の価格差が可視化 |
| 偽物・注意点の情報が探しにくい | アイテムごとのコミュニティ知見 |
| 自分のコレクション管理ができない | 所持品管理+相場参照が一体化 |

### 9-4. コアエンゲージメントループ（Hookedモデル）

```
[トリガー] 仕入れた/買った →
[アクション] アプリで写真+価格を記録（30秒以内） →
[可変リワード] 他ユーザーの反応、バッジ獲得、相場データの更新 →
[投資] データが蓄積され、自分のコレクション履歴が豊かに →
[トリガーに戻る] 次に何か買った時にまた記録したくなる
```

### 9-5. 収益化前のサステナビリティ

**Discogsモデル: 5年間収益ゼロでも維持できた理由:**
1. 創業者が本業の傍らで運営（さっさ=せどりが本業）
2. インフラコストが極めて低い（初期は自宅サーバー）
3. モデレーションはコミュニティに委任

**本プラットフォームの場合:**
- Phase 1-2（0-18ヶ月）のコスト: 月150-7,000円（Supabase + R2 + Vercel）
- さっさのせどり本業が存在するため、プラットフォーム収益に依存しない
- 戦略A（note/Threads）の収益で開発期間の生活費を補填
- **「収益化を急がない」ことがプラットフォームの信頼を築く最大の武器**

### 9-6. 競合詳細分析

**オークファン（3674.T）の弱点:**
- **2025年9月期に赤字転落（営業▲1.9億円）**。事業縮小中（NETSEAオークション終了、Amacode譲渡）
- 有料会員わずか**3.1万人**（無料107.9万人の2.9%）→ マネタイズ失敗
- 2024年7月に2倍超値上げ（998円→2,200円）→ ユーザー反発
- メルカリ連携は2024年5月開始だが過去データが浅い。Pro Plusメルカリ検索は**廃止済み**
- まんだらけ、駿河屋、らしんばん等の専門店価格は非対応
- UIが2010年代のまま。コミュニティ機能なし
- **競合として弱体化中。脅威度は低い**

**MyFigureCollection（MFC）の弱点:**
- **トラフィック37%減（2025/8→2026/1: 3.33M→2.1M/月）**。ユーザーが減少を指摘
- UIの頻繁な変更が改悪と認識されている
- コレクション管理がスプレッドシート表示のみ（画像表示なし）
- サーバーダウンタイムが多い（バンドエイド修正の繰り返し）
- モバイルアプリが事実上死亡（2012年のAndroidアプリが最後）
- **中国Hpoiに機能面で圧倒されている**（ユーザー自身が「Hpoiのほうが遥かに有用」と認めている）
- マーケットプレイスの信頼崩壊（PayPal F&F詐欺、ゴースティング、ネガティブレビュー削除疑惑）
- r/AnimeFigures（Reddit）のほうがソーシャル体験として優れているとの声
- **相場データが事実上ゼロ**: 自動価格追跡なし、履歴グラフなし
- **フィギュア以外（トレカ、アパレル、一番くじ、イベント限定グッズ等）をカバーしない**
- DB登録数85万件。しかしアクリルスタンド、ぬいぐるみ、一番くじ等は穴だらけ

**monoca（コレクション管理アプリ）の弱点:**
- 無料100アイテム制限
- 相場データなし（個人の記録のみ）
- コミュニティなし（個人完結）
- 大規模コレクションでパフォーマンス低下

→ **すべての競合に共通する最大の弱点: 「コレクション管理」と「相場データ」と「コミュニティ」が統合されていない**

### 9-7. アニメコレクターコミュニティの実態

| コミュニティ | 規模 | 特徴 |
|------------|------|------|
| r/AnimeFigures（Reddit） | 182K+メンバー | 写真共有・質問・購入相談が活発。MFCより良いソーシャル体験 |
| MFC | 「フィギュアコレクティングのMySpace」と呼ばれ始めている | 技術的停滞、ユーザー離れ加速 |
| Hpoi（中国） | 急成長 | 360度ビュー、モダンUI。MFCを機能面で完全に圧倒 |
| @figure_maniax等（X/日本） | 万単位フォロワー | 日本コレクターの情報源はXとまんだらけ |

**最大の未解決課題（＝本プラットフォームの勝ち筋）:**
1. **アフターマーケット価格の不透明性**: 全コレクター共通のペインポイント
2. **日本と欧米コミュニティの分断**: 橋渡しプラットフォームが存在しない
3. **フィギュア以外のアニメグッズDBが皆無**: アクスタ、缶バッジ、一番くじ等
4. **コレクション管理の標準ツール不在**: スプレッドシートかMFCの貧弱リスト

### 9-8. ブランド戦略

プラットフォームにはさっさとは独立したブランドを付ける。

- さっさの個人ブランドとは分離
- 理由: プラットフォームの中立性・信頼性が重要。個人ブランドだと「宣伝ツール」に見える
- さっさは「創業者/メインコントリビューター」として裏方で動く
- 海外展開の余地を残すため、英語でも通じる名前が望ましい
