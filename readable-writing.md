# 読みやすさ設計スキル

**他のスキル（sassa-writing, wp-sedori, wp-article-structure等）と併用する下地スキル。**
文体やトーンは各スキルに従い、「読みやすさ」の構造設計はこのスキルに従う。

---

## 1. 構成設計（最重要）

読みにくい記事の最大原因は文の長さではなく**構成**。情報の並び方・階層・密度が悪いと、どれだけ1文を短くしても読めない。

### 1-1. 目次で殺すな

- **H2は5〜8個が上限**（目次に出る。10個超えたら読者は目次で離脱する）
- H3で分割すれば情報量は維持できる。H2を増やすな、H3で整理しろ
- 目次だけ読んで「何が書いてあるか」「読む価値があるか」が3秒でわかること

### 1-2. 構成の型

#### 短い記事（2,000〜4,000文字）
```
H2 × 3〜5
各H2の下にH3 × 2〜3
```

#### 長い記事（5,000〜10,000文字）
```
H2 × 5〜7
各H2の下にH3 × 2〜4
H4は実売事例の商品名のみ
```

#### 有料長編記事（10,000文字超）
```
無料: H2 × 3〜4（全体の25〜35%）
有料: H2 × 4〜6
各H2の下にH3で整理。H2の乱立は絶対禁止
```

### 1-3. 有料記事のペイウォール設計

#### 核心原則: What無料 / How有料

**「何をすべきか（What）」は無料で出す。「具体的にどうやるか（How）」を有料にする。**

これがNYT・日経・noteで共通する有料コンテンツの設計原則。SEO流入と有料転換を両立させる唯一の方法でもある。

| | 無料で出す（What） | 有料にする（How） |
|---|---|---|
| フレームワーク | 「3つの基準がある」と名前・概要を出す | 各基準の具体的な使い方、判断プロセス |
| 事例 | 2〜3件チラ見せ（数字だけ） | 全事例の思考過程・判断ロジックを完全公開 |
| テクニック | 「こういう方法がある」と存在を示す | ステップバイステップの手順 |
| 失敗 | 「こういう失敗がある」と警告 | 失敗の詳細分析・回避方法 |

**なぜこれが効くか:**
- 無料部分がSEOキーワードを含む実質的な解説になる → 検索上位を取れる
- 読者は「Whatはわかった。じゃあHowは？」と自然に有料部分を欲しがる
- 出し惜しみ感が出ない（Whatは惜しみなく出してるから）
- 無料部分だけ読んでも「読んでよかった」と思える → 信頼構築

**アンチパターン:**
- ❌ Whatも隠す → 出し惜しみ感。「結局何が書いてあるかわからない」で離脱
- ❌ Howまで無料で出す → 有料部分を買う理由がなくなる
- ❌ 「答えの輪郭だけ見せる」→ 読者は見透かす。SEOにも弱い

#### SEO + 有料コンテンツの両立（WordPress + codoc）

codocはクライアントサイドJS制御 → **Googlebotには有料部分も全文見える**。つまりSEO的には有料部分もインデックスされる。

ただしこれを正当化するために**構造化データが必要**:
```json
{
  "@type": "Article",
  "isAccessibleForFree": false,
  "hasPart": {
    "@type": "WebPageElement",
    "isAccessibleForFree": false,
    "cssSelector": ".codoc-entries"
  }
}
```

これを入れないとGoogleに「クローキング（スパム）」と判定されるリスクがある。

#### 無料部分の構成（SEO記事型）

noteの有料記事とは構成が違う。**前半がSEO記事として成立する必要がある**。

```
[無料] H2-1: 検索意図を満たす解説（What）
  → SEOのメインターゲット。このセクションだけで検索上位を狙える内容
  → フレームワークの名前・概要・なぜ効くかを出す

[無料] H2-2: 実例チラ見せ + 信頼構築
  → 2〜3件の事例（数字だけ見せる。判断プロセスは有料）
  → 筆者の実績・合計利益を提示

[無料] H2-3: よくある疑問 + 有料プレビュー
  → Q&Aで読者の不安を解消（SEOにも効く）
  → 有料部分の目次・得られるものを提示
  → ペイウォール

--- ペイウォール ---

[有料] 即効性のあるノウハウ1つ → 深掘り → 事例群 → 応用 → まとめ
```

**無料3〜4セクション / 有料4〜6セクション**。無料は全体の25〜35%。

#### ペイウォール直前の鉄則

- **Whatを出し切った直後にペイウォール**（読者が「じゃあ具体的にどうやるの？」と思った瞬間）
- **最もインパクトのある数字を無料で出す**（「一撃16,000円利益」→ これがSEOにも効く）
- **無料部分だけでも「読んでよかった」と思える価値がある**

#### 有料部分の「即効性→深掘り→応用」3段階

```
Stage 1: 即効性（20%）
  → 読んですぐ使える具体ノウハウを1つ出す
  → 「買ってよかった」の第一印象をここで確定させる

Stage 2: 体系的な深掘り（60%）
  → フレームワークのHow、事例群、構造解説
  → メインコンテンツ

Stage 3: 応用・まとめ（20%）
  → 読者が自分の状況に当てはめるガイド
  → チェックリスト、テンプレートなど「持ち帰れるもの」
```

### 1-4. 見出しの書き方

- **15〜28文字**。30文字超えたら削れ
- 見出しに「メリット」がわかる言葉を入れる（「コラボ系アクスタ」→「コラボ系 — 見つけたら即チェック」）
- 抽象的な見出し禁止（「ポイント」「大切なこと」「注意点」→ 具体的に何の話か書け）

## 2. 読者を引っ張る構造設計

### 2-1. オープンループ（ツァイガルニク効果）

脳は「未解決の問い」を閉じたがる。セクション末に問いを開き、次セクション冒頭で回収する。これが読み進める推進力になる。

**ループの5層構造:**

| レベル | スコープ | 例 |
|--------|----------|-----|
| **記事ループ** | 記事全体を貫く1つの問い | タイトルで提示→まとめで回収 |
| **セクションループ** | H2単位の問い→回答 | セクション冒頭で問い→セクション内で閉じる |
| **段落ループ** | 数段落の問い→回答 | 「実はこれ、ある条件を満たすと化ける。」→次の段落で条件を開示 |
| **一文ループ** | 1文の問い→次の文で回答 | 「結論から言うと——」の直前 |
| **方向転換ループ** | 接続詞1語で予告 | 「しかし、ここに落とし穴がある。」 |

**設計ルール:**
- 開いたループは**1〜3段落以内**に閉じろ。長すぎると苛立ちに変わる
- 煽って中身スカスカは最悪。開いた問いの強度と、閉じるときの報酬の強度を一致させろ
- セクション末で新しいループを開き→次セクション頭で前のを閉じながら新しい問いへ→**チェーン構造**

**実装パターン:**
```
【H2: 仕入れ値を半額にする方法】
（内容）
...ただし、これだけでは利益は出ない。「売り方」に致命的な罠がある。
                                    ← ループを開く
【H2: 9割が間違える売り方の落とし穴】  ← 次セクションで回収
実は先ほどの仕入れ法で得た商品を、そのまま出品すると...
```

### 2-2. テンションカーブ（緊張と弛緩のリズム）

記事全体に「緊張→弛緩」の波を作る。常に緊張が高いと疲れる。常に低いと飽きる。

```
緊張度
 高 |    /\        /\
    |   /  \  /\  /  \
    |  /    \/  \/    \___
 低 |/
    +------------------------→ 記事の進行
     導入  中盤      結末
```

| 要素 | 緊張を上げる | 弛緩させる |
|------|-------------|-----------|
| 内容 | 問い・対立・意外な事実・数字 | 解説・具体例・体験談・共感 |
| 文体 | 短文・断定・体言止め | 柔らかい語尾・吹き出し・雑談 |
| 視覚 | 太字・マーカー・テーブル（データ） | 画像・余白・吹き出し |

**設計方法:**
- 2〜3段落の「密な情報」の後に、1段落の「軽い話」を入れる
- H2の前半で緊張（問題提起・データ）→後半で弛緩（具体例・体験談）
- 記事の最後は弛緩で終わる（読後感が良くなる）

**文章レベルでの緩急の実装:**
- **緩（スローダウン）**: 長めの文、描写中心、改行少なめ。感情・思考を丁寧に書く
- **急（スピードアップ）**: 短文、事実の羅列、体言止め、改行多め、会話文
- **緩→急の切り替え**: 段落末に短い一文を置く（「で、結果どうなったか。」）
- **急→緩の切り替え**: 場面描写から入る（「翌朝、メルカリの通知が鳴った。」）

### 2-4. マイクロフック（注意を引き戻す仕掛け）

人の集中力は約10分（文章では約2,000〜3,000字）で低下する。定期的にフックを挿入しろ。

| フックの種類 | 例 | 挿入タイミング |
|---|---|---|
| **問いかけ** | 「この商品、あなたならいくらで仕入れる？」 | 事例解説の直前 |
| **意外な事実** | 「実はこの商品、99%の人がスルーする」 | 長い解説の途中 |
| **自分ごと化** | 「あなたの家に、同じカテゴリのもの、ないですか？」 | 3事例ごと |
| **予告** | 「次の商品が今回の記事で一番利益率が高い」 | セクション末尾 |
| **数字の衝撃** | 「仕入れ300円→販売4,800円」 | 新セクション冒頭 |
| **失敗談** | 「俺はこれで3万円溶かした」 | 読者が慣れてきた中盤 |

**2,000〜3,000字ごとにフックを1つ入れろ。**

### 2-3. 初頭効果・新近効果（系列位置効果）

人は最初と最後に触れた情報を最もよく記憶する。中間は忘れる。

| 位置 | 置くべき情報 | 理由 |
|------|-------------|------|
| **記事冒頭** | 最も重要な主張・結論 | 初頭効果。長期記憶に刻まれる |
| **各セクション冒頭** | そのセクションの核心 | セクション内でも初頭効果が働く |
| **記事末尾** | CTA・まとめ・印象的な一言 | 新近効果。行動に直結する |
| **中間部** | 補足データ・例外・詳細 | 記憶に残りにくいが文脈理解には必要 |

**リスト型記事での応用:** 最強の商品・事例を1番目と最後に置く。中間に「まあまあ」のものを配置。

## 3. 認知負荷のコントロール

### 3-1. 認知負荷の3分類

| 種類 | 意味 | 対策 |
|------|------|------|
| **課題内在性負荷** | 内容自体の難しさ | 前提知識を先に補う。専門用語をいきなり出さない |
| **課題外在性負荷** | 見せ方による無駄な負荷 | レイアウト整理、余白、視覚階層。**これを徹底的に削る** |
| **学習関連負荷** | 理解を深めるための良い負荷 | 具体例・比較・「自分ならどうする？」の問い。**これは意図的に入れる** |

### 3-2. チャンキング（情報のグループ化）

- 人のワーキングメモリは**3〜6チャンク**が限界
- 1段落に情報を詰め込むな。意味の塊ごとに分けろ
- 長いセクションの冒頭に**サマリー段落**（このセクションの結論を1〜2文で）を置く
- 背景色・罫線・余白で「関連グループ」を視覚的に分離する

### 3-3. 段階的開示（Progressive Disclosure）

今この瞬間に必要な情報だけ見せ、詳細は後から。

| パターン | やり方 | 例 |
|----------|--------|-----|
| **結論ファースト + 詳細後置** | 各セクション冒頭に結論→詳しく知りたい人だけ読み進める | H2直下に太字で結論→その下に根拠と具体例 |
| **概要マップ提示** | 記事冒頭で全体像を箇条書き | 目次だけでなく「3つの要点」として要約 |
| **抽象→具体の段階展開** | まず概念を1文→次に具体例→最後にケーススタディ | 「要するに○○だ」→「例えば〜」→「実際に僕が〜」 |

### 3-4. 抽象⇔具体のハシゴ

```
抽象度 高  ← 「なぜ」「意味」「原則」
    ↑      「せどりは"ズレ"を利益に変える行為だ」
    |
    |      「仕入れ先と販売先の価格差がズレである」
    |
    ↓      「オフモールで1,200円の商品がメルカリで3,500円で売れた」
抽象度 低  ← 「いつ」「どこで」「いくら」
```

**ルール:**
- 同じ抽象度が3段落以上続いたら飽きる。具体→具体→具体が続いたら一度引いて抽象に戻す
- **サンドイッチ構造**: 抽象（原則）→具体（事例）→抽象（まとめ・応用）
- **ズームイン/ズームアウト**: セクション前半で全体像→後半で1つの事例に深く入る→次セクション冒頭で再び引いて全体像に戻す

## 4. 繰り返し構造の処理（カテゴリ別・商品別）

同じフォーマットの繰り返しは読者を飽きさせる最大の罠。

### 4-1. 基本ルール

- **1つのH2の中にH3で並べる**（H2を5個に分けるな）
- テーブルや箇条書きで圧縮できる部分は圧縮する

### 4-2. 情報密度の波（「詳→圧→圧→詳」パターン）

全部同じ密度で書くな。**波を作る。**

```
事例A: ★★★★★（フル解説。思考過程・相場調査・利益計算を全部見せる → 読者に「型」を教える）
事例B: ★★★（圧縮。Aで教えた型を前提に、差分だけ解説）
事例C: ★★（さらに圧縮。Bとの違いだけ）
--- ここでマイクロフック or ミニストーリー挿入 ---
事例D: ★★★★（フル解説に戻す。意外な切り口で注意を引き戻す）
事例E: ★★（簡潔に締め）
```

### 4-3. 入り口の変奏

基本構造は同じでも、情報の入り口を変える:
- 1つ目: **数字から入る**（「利益率120%。アクスタの中でも異常値だ」）
- 2つ目: **エピソードから入る**（「これを見つけたのはブックオフの隅の棚だった」）
- 3つ目: **読者への問いかけから入る**（「このアクスタ、いくらで売れたと思う？」）
- 4つ目: **意外な事実から入る**（「実はこのカテゴリ、利益がほぼ出ない」）

### 4-4. 視点シフトで変化をつける

同じ構造の繰り返しでも、視点を変えるだけで別物に見える:
- 事例A: **仕入れ目線**で解説（どう見つけるか）
- 事例B: **販売目線**で解説（どう売るか）
- 事例C: **失敗目線**で解説（やりがちなミス）
- 事例D: **数字目線**で解説（利益率・回転率の分析）

### 4-5. 中間まとめの挿入

3〜4個紹介したら**ブリッジ**を挟む:
- 「ここまでの共通点は〜」と中間まとめ
- 視点を切り替えてから次のグループへ
- これで読者に「今どこにいるか」を再確認させる

### 4-6. ミニストーリーの挿入

事例が3つ続いたら、場面転換として30秒で読める小話を挟む:
```
事例A解説
事例B解説
--- 場面転換 ---
[コラム] 「実はこの商品B、仕入れた当日に面白いことがあって...」
--- 解説に戻る ---
事例C解説
```

### 4-7. フォーマットの切り替え

テキスト→テーブル→箇条書き→テキスト→吹き出し。視覚フォーマットを変えるだけでリズムが変わる。同じフォーマットが3回以上連続しないようにする。

## 5. 読者のナビゲーション（迷子にさせない）

### 5-1. 現在地の提示

- 見出しの階層（H2/H3/H4）で「今の深さ」を視覚的に示す
- 繰り返し構造では「5つのカテゴリのうち3つ目」のような進捗を示す
- 長いセクションの冒頭で「このセクションでは○○と△△を扱う」と地図を見せる（ただし構造宣言臭くならないように自然に）

### 5-2. 読者の反論を先回り

「こう思うかもしれないが〜」と読者の心の声を代弁してから答える。これ自体がナビゲーションになる。読者は「この筆者は自分の疑問をわかってる」と感じて読み続ける。

### 5-3. 戦略的反復

重要なポイントは言い方を変えて2〜3回繰り返す。初出で概念を導入→中盤で具体例とともに再登場→まとめで結論として再提示。迷子になった読者の再合流ポイントになる。

## 6. 場面転換の技法

映画脚本から借りた5つの転換パターン:

| パターン | 文章での実装 |
|---|---|
| **カットイン** | 解説の途中で突然エピソードを挟む。「ところでこの商品、仕入れた日に面白いことがあった」 |
| **タイムスキップ** | 「3ヶ月後」「翌週の結果を先に言うと」で時間を飛ばす |
| **視点切り替え** | 著者目線→読者目線、仕入れ側→販売側 |
| **対比挿入** | 「一方で、失敗した例も見てみよう」 |
| **ブリッジ** | 前セクションの最後の言葉を次の頭で拾う。「利益率の話をした。では、利益率が"嘘をつく"パターンを見る」 |

## 7. 1万字超の長文構成フレームワーク

### フレームワークA: 「導入→基礎→事例群→応用→締め」（実用ノウハウ型）

```
[1] 導入（1,000字）
    フック + 記事の価値提示 + 読者像の明示

[2] 基礎フレームワーク（2,000字）
    思考の「型」を1つ教える。この型で以降の事例を読み解ける、と予告

[3] 事例群（5,000〜7,000字）
    事例1: フル解説（型の適用を丁寧に見せる）
    事例2: 圧縮（差分だけ）
    [マイクロフック挿入]
    事例3: 圧縮（別パターン）
    事例4: フル解説に戻す（意外な切り口）
    [コラム: 失敗談 or 裏話]
    事例5: 圧縮

[4] 応用・自分ごと化（1,500字）
    読者が自分の状況に当てはめるガイド。チェックリスト or テンプレート

[5] 締め（500字）
    全体の要約（型の再提示）。行動の呼びかけ
```

### フレームワークB: 「問題→失敗→発見→体系→実践」（ストーリー型）

```
[1] 問題提起（1,000字）: 読者が共感する「あるある」の状況
[2] 失敗フェーズ（1,500字）: 著者の試行錯誤。「こうやって失敗した」
[3] 発見フェーズ（1,500字）: 転機。何がきっかけで変わったか
[4] 体系化フェーズ（4,000字）: 発見を整理。フレームワーク化
[5] 実践フェーズ（2,000字）: 具体的な事例で体系を証明
[6] 読者の番（1,000字）: テンプレート、チェックリスト、次のステップ
```

### 章の開始パターン（単調さ回避）

| パターン | 例 |
|---|---|
| 数字で始める | 「利益率78%。これが3つ目の商品の実力だ」 |
| 問いかけで始める | 「なぜ、この商品をスルーする人が多いのか？」 |
| 場面描写で始める | 「オフモールの画面をスクロールしていた時、手が止まった」 |
| 結論から始める | 「結論から言うと、この方法が一番再現性が高い」 |

## 8. 教材設計フレームワークの応用

### ガニェの9教授事象（有料記事に変換）

## 9. セクション設計

### 9-1. H2セクションの長さ

- **300〜800文字が適正**（H3込み）
- 1,000文字を超えたらH3で分割するか、H2を分けることを検討

### 9-2. H3セクションの長さ

- **150〜400文字が適正**

### 9-3. セクションの冒頭

- 各H2の直後に**1〜2文の導入**を入れる
- いきなり詳細に入るな。ただし「このセクションでは〜を解説します」はAI臭い。自然にやれ
- **結論ファースト**: そのセクションで最も重要な主張を最初に置く

### 9-4. セクション間のつなぎ

- 前のセクションの結論を受けて次に入る（1文でいい）
- オープンループで繋ぐのが最も自然（2-1参照）
- 機械的な接続詞（「では次に」「続いて」）の連発はNG
- つなぎが不要な場合は入れない。無理につなげるな

## 10. 段落設計

### 10-1. 段落の長さ

- **1段落 = 1〜3文**
- **スマホ表示で5行を超えたらアウト**（PC3行 ≒ スマホ5〜7行）
- 文字数目安: **1段落150文字以内**

### 10-2. 段落の役割

- **1段落 = 1トピック**
- 段落の1文目にそのトピックの結論を書く（読者は1文目だけ読んで飛ばし読みする前提）
- 残りの文は補足・具体例

### 10-3. 改行ルール

- 句点「。」で改行。文の途中で改行しない
- 段落間に空行を入れる
- 1文ごとに空行はスカスカ。2〜3文でまとめてから空行

---

## 11. 文の設計

### 11-1. 文の長さ

- **1文 = 40〜60文字が目安**
- **80文字超は即分割**
- 読点が3つ以上ある文 → 長すぎるシグナル。分割しろ

### 11-2. 一文一義

- 1文に主語は1つ。情報は1つ
- 接続助詞（「〜が」「〜ので」「〜けど」「〜して」）で文をつなげるな。そこで切れ

### 11-3. 読点の打ち方

- 1文に読点は最大2つ。できれば1つ
- 20文字以内ごとに1つが目安
- 誤読リスクがある箇所には必ず打つ

### 11-4. 文末バリエーション

- **同じ語尾は最大2回連続。3回連続は禁止**
- バリエーション: です/ます/でした/ました/でしょう/ください/ですよね/ません/体言止め/疑問形

### 11-5. 漢字・ひらがなバランス

- **漢字3割 : ひらがな7割**
- 漢字が連続すると圧迫感、ひらがなが連続すると区切り不明

## 12. 視覚リズム

### 12-1. 視覚要素の配置間隔

- **300〜500文字ごとに視覚的な変化を入れる**
- 視覚要素: 画像 / 箇条書き / テーブル / ボックス / 吹き出し
- テキストだけが3スクロール以上続いたら危険信号

### 12-2. 画像

- H2見出しの直後に1枚（箸休め。セクション切り替わりの視覚的マーカー）
- ボックス・箇条書き・テーブルが多いセクションなら画像なしでも可

### 12-3. 箇条書き

- **3つ以上の列挙は必ず箇条書きにする**
- 1つの箇条書きの項目数: 3〜7個
- 各項目は1〜2行で収める

### 12-4. 装飾の密度

- 太字: 1段落に1箇所まで。**太字だけ拾い読みして要点がわかる状態**が理想
- マーカー: 1セクション（H2区間）に1〜2箇所
- 色数: 装飾に使う色は2〜3色まで
- **装飾なし段落 : 装飾あり = 7 : 3**

### 12-5. 吹き出し

- 短い一言で使う。長文を入れるな
- 連続使用禁止
- 1セクション（H2区間）に0〜1個

## 13. スマホファースト

- 閲覧の70〜80%はスマホ。**スマホ基準で設計する**
- フォントサイズ: 16px以上
- 行間: 1.5〜1.8em
- **公開前にスマホ実機でプレビュー確認は必須**
- PCで3行 = スマホで5〜7行。PC基準で書くとスマホで詰まる
